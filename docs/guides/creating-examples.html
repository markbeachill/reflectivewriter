Creating safe worked examples

How to turn a reflective-writing tutoring exchange into a public or local teaching example without exposing private material or showing the AI writing the reflection for the student.

Back to guidesView worked examplesDeployment check

The core rule

A Reflective Writer example must show the AI teaching a reflective move — diagnosing, questioning, explaining, or setting a next task. It must not show the AI producing a finished reflection, inventing the writer’s experience, supplying feelings the writer has not named, or deciding what the writer learned.

Use fictional or composite material by default

The safest examples are fictional or composite. They can still feel realistic: use common reflective-writing problems such as over-description, vague learning, box-ticking a model, blame-heavy tone, or missing action planning. Avoid copying a real student’s wording unless you have explicit permission and a clear local reason to do so.

Healthcare, placement and workplace examples need extra care. Remove not only names, but also dates, locations, rare events, job titles, team details and combinations of facts that could identify a person to someone who knows the setting.

The five-step workflow

  1. Choose the teaching point. Decide what the example should demonstrate: spotting description, deepening the “so what”, checking anonymity, resisting ghost-writing, or another specific move.
  2. Create or collect the exchange. Use invented or composite student text where possible. If starting from a real session, strip it back to the teaching pattern rather than preserving the student’s personal story.
  3. Anonymise manually. Remove direct identifiers and jigsaw identifiers. Do not rely on the AI to decide that an example is safe to publish.
  4. Check the authorship boundary. The AI’s reply should ask the student to write the next sentence or decision in their own words. It should not provide a polished paragraph for submission.
  5. Store the source record and rebuild. Add or update the YAML file in src/examples/items/, then run python scripts/build_examples.py.

What every public example source must include

Each public example is generated from a YAML source record. The source stores the tool code, the teaching context, a transcript, a privacy note and explicit checks. Public examples must pass all checks before being rendered.

id: rf1-description-detox
code: RF1
tool_id: description-detox
status: public
safety_level: fictional
context: How RF1 separates retelling from reflection.
learning_point: How RF1 separates retelling from reflection.
privacy_note: Fictional or composite teaching example; no real person is identifiable.
checks:
  authorship_boundary: true
  no_identifying_detail: true
  unrelated_made_up_example_only: true
  no_finished_reflection: true
  privacy_reviewed: true

See src/examples/example-schema.md in the repository for the full schema.

Public, local and internal examples

Public examples are suitable for the website and should normally be fictional or composite. Local examples may refer to a course task, but should still avoid identifiable people and should not contain a student’s private reflection unless local consent and policy requirements are met. Internal examples can be used for maintainer testing, but they should still avoid unnecessary personal or sensitive data.

Do not publish

Do not publish an example if it contains a real patient, client, service user, colleague, placement provider, institution or student who could be recognised; if the AI writes the reflective paragraph for the student; or if the example depends on feelings or learning that the student did not actually supply.

Using a completed session as source material

A completed tutoring session can be useful evidence of how a tool behaves, but treat it as raw material, not as a publishable example. Ask what the session demonstrates, rewrite private or sensitive content into fictional form, and keep only the pattern of tutoring. If you ask an AI to summarise a session into a possible example, check the summary manually: the AI may omit privacy risks, overstate compliance, or smooth over moments where it did too much writing.

Final review questions

  • Could anyone identify a real person or place from direct or combined detail?
  • Does the AI preserve the writer’s authorship of experience, feeling, insight and learning?
  • Is the teaching point clear from the example?
  • Would this model good AI use to a student?
  • Have the YAML checks been completed honestly?
